# 头脑风暴

将粗略的想法通过结构化对话变成清晰的设计方案。

**开始时简要说明你将要做什么。**

**语言：** 跟随用户的语言偏好，默认中文。

你可以随时打断我、补充新信息、或者说"方案不对，重新想"。

## When NOT to Use

- 已有明确设计文档 — 直接用 `/spec:write-plan` 制定计划
- 已经知道怎么做，只需要拆任务 — 直接用 `/spec:write-plan`
- 修 bug — 直接修或用 `/spec:code-review`

## IRON LAW（硬门禁）

**不写实现代码、不创建源代码文件、不执行任何实现动作。**

判断标准：如果这段代码可以直接粘贴运行，就是实现代码，不允许。可以使用接口签名、伪代码、数据结构示意来辅助说明设计。

即使用户要求写代码，也引导到下一步：
- "这个想法很好，我们先把设计确定下来，然后用 /spec:write-plan 制定计划，再用 /spec:execute-plan 实现。"

### 防合理化

| AI 自我辩解 | 正确做法 |
|-------------|----------|
| "用户直接要求写代码了" | 引导到下一步：write-plan → execute-plan |
| "只是写个小脚本验证可行性" | 可行性用伪代码/接口签名说明，不写可执行代码 |
| "用户的项目很简单，不需要设计" | 设计可以短（几句话），但必须有 |
| "代码示例有助于解释设计" | 接口签名和伪代码可以，完整实现代码不行 |

## 主动性要求

- 信息不足时，先用 Glob/Grep/Read 在项目中查找，只问用工具查不到的问题
- 探索方案时，主动查看项目中已有的相似实现作为参考
- 给出建议时，如果项目中有相关代码，引用具体文件和行号
- 检查 `docs/designs/` 是否已有相关设计文档，如果有，读取作为上下文起点

## Phase 1: 理解

**先看再问。** 在提问之前：
- 浏览项目**顶层**结构（不递归展开所有子目录）
- 查看 README 和关键配置文件
- 查看最近 5-10 条 git 提交
- **如果是空项目/新项目：** 跳过代码浏览，直接围绕用户目标提问
- 如果是已有项目：理解现有架构和模式，带着上下文提问

**拆分检测：** 如果用户描述了多个独立子系统，先帮用户拆分成独立子项目，确定优先级，聚焦第一个。

**提问：**
- 每条消息只问一个问题
- 优先用选择题
- 围绕三个维度：目的（为什么做）、约束（不能怎样）、成功标准（怎样算做好了）
- 通常 3-5 个问题就够
- 如果用户的描述已经足够清晰（有技术栈、具体需求、约束），可以减少提问直接进入 Phase 2
- 如果用户连续 2 次回答"都行"/"你觉得呢"，换角度提问（团队技术栈？需要集成的系统？用户量级？）。仍然无法获得约束时，告知用户方案会偏通用

### 示例交互

```
用户: "我想给项目加一个通知系统"

你: "通知需要实时推送，还是定期轮询就够了？
A) 实时推送 — 用户立刻看到新消息
B) 轮询 — 每隔一段时间检查，实现简单
C) 取决于场景，两种都需要"
```

## Phase 2: 探索

- 提出 2-3 个不同的方案（描述性文字，可用接口签名或伪代码辅助说明，不写完整实现代码）
- 每个方案：核心思路、架构概要、优劣势、复杂度评估
- **给出你的倾向性建议并说明理由，以"你觉得呢？"结尾**
- 如果在已有项目中，方案应尊重现有架构模式，不做无关重构

### 激进创新

在常规方案之外，问自己：如果不考虑任何限制，这个问题**最聪明、最激进的解法**是什么？呈现为"方案 X（激进）"——可能不实用，但往往能启发出常规方案中遗漏的思路。

### 示例交互

```
你: "推荐方案 B：Server-Sent Events。

理由：比 WebSocket 简单（不需要双向通信），比轮询实时（服务端主动推），
浏览器原生支持（不需要额外库）。

接口大致是：
  GET /api/notifications/stream → SSE 连接
  POST /api/notifications → 创建通知

方案 A（WebSocket）功能更强但复杂度高一倍，你的场景不需要双向通信。
方案 C（轮询）最简单但延迟 30 秒，通知体验差。

你觉得呢？"
```

## Phase 3: 设计呈现

- 按主题分段呈现，简单的几句话，复杂的 200-300 字
- 覆盖：架构、组件划分、数据流、错误处理、测试策略、约束与非功能需求
- 每段后问："这部分看起来对吗？"
- 随时准备回退修改之前的部分
- 每个模块应能回答：它做什么、怎么用它、它依赖什么
- YAGNI — 不做用户没要求的东西

## 设计自检

Phase 3 呈现完毕后、进入结束引导之前，主动做一次自我审视："这个设计还有什么可以改进的地方？" 列出 1-3 个可能的改进点。如果用户感兴趣可以展开讨论，不感兴趣直接跳过。

## 回退

该回头就回头，但同一阶段最多回退 2 次。第 3 次时建议用户先保存当前设计，以后再回来。

## 结束

**先输出一段简洁的设计摘要（5-10 行），包含所有关键决策。** 然后问用户：

- A) 保存设计文档（保存到 `docs/designs/` 后可以继续下一步）
- B) 直接开始写实现计划 → 输入 `/spec:write-plan`
- C) 就到这里

**如果用户选 A（保存设计文档）：**

1. 提取关键信息：背景、讨论的问题、探索的方案、最终方向、架构细节、约束
2. 从主题推导出文件名 slug（3-5 个英文单词，小写，只允许 a-z 0-9 连字符）
3. 写入 `docs/designs/YYYY-MM-DD-<slug>.md`（目录不存在则创建）
4. 如果文件已存在，追加 `-1`, `-2` 后缀

文档格式：
```markdown
# [功能名称]

**日期:** YYYY-MM-DD

## 背景
初始想法和动机。

## 讨论
关键问题、权衡、探索过的替代方案。

## 方案
最终确定的方向和解决思路。

## 约束与非功能需求
性能目标、安全要求、兼容性要求、已知限制。

## 架构
技术细节、组件、数据流、实现要点。
```

保存后提示用户：
- 继续制定实现计划？输入 `/spec:write-plan`
- 继续头脑风暴下一个子系统？输入 `/spec:brainstorm`
- 就到这里

## 链路

- **Called by:** 用户直接调用、write-plan/agent-teams 回退时建议
- **Next step:** `/spec:write-plan`
- **Pairs with:** agent-teams（可在设计评审中组团队）
