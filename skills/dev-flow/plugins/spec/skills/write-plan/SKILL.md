# 编写实现计划

将设计文档或需求描述拆解成详细的、可逐步执行的实现计划。

**开始时简要说明你将要做什么。**

**语言：** 跟随用户的语言偏好，默认中文。

## When NOT to Use

- 需求还不清晰 — 先用 `/spec:brainstorm` 探索设计
- 小 bug 修复或单文件改动 — 不需要计划，直接改
- 纯重构且范围明确 — 直接改或用 `/spec:code-review` 审查

## IRON LAW（违反即停止）

1. **每个 Task 必须有验证步骤**（运行命令 + 预期输出）— 没有验证 = 不完整
2. **文件路径、API、依赖必须用工具确认存在** — 不凭记忆猜测
3. **每个代码修改必须可追溯到设计文档或需求** — 无来源的修改不写入计划

### 防合理化

| AI 自我辩解 | 正确做法 |
|-------------|----------|
| "这个路径我记得是对的" | IRON LAW 要求：用工具确认存在。grep 一下 |
| "这个任务太简单不需要验证步骤" | 每个 Task 必须有验证步骤，是 IRON LAW 要求 |
| "设计文档有歧义但我能猜到意思" | 先查代码推断，查不出就问用户。不猜 |
| "TDD 对这个任务不合适" | 已有非 TDD 模式。选模式不是跳过验证的理由 |

## 主动性要求

- 拆分任务前，主动用 Glob/Grep 确认计划中涉及的文件确实存在、路径正确
- 写测试代码前，主动查看项目已有的测试风格并遵循
- 发现设计文档有歧义时，先查项目代码看是否能推断出意图，只问用工具查不到的问题

## 1. 启动检查

```bash
git branch --show-current
ls docs/designs/ 2>/dev/null || echo "NO_DESIGNS_DIR"
ls docs/plans/ 2>/dev/null || echo "NO_PLANS_DIR"
```

## 2. 加载设计

- 如果用户指定了设计文档路径，读取它
- 如果没指定，检查 `docs/designs/` 下的文件：
  - 只有一个文件 → 使用它并告知用户
  - **有多个文件 → 列出所有文件让用户选择，不自动选**
- 如果没有设计文档 → 基于对话内容或用户口头描述制定计划（跳过 brainstorm 也可以直接用）
- 检查最近 git 提交，确认代码库状态

## 3. 审查设计

- 仔细阅读设计文档，浏览相关代码文件
- 如果设计有模糊、矛盾或缺失 → 列出问题等用户澄清
- 如果问题太多或太根本 → 建议用户先完善设计：输入 `/spec:brainstorm`
- 不猜测，不自行填补设计缺口

## 4. 项目偏好确认

在拆分任务之前，检测项目是否有测试框架。如果没有，问用户：
- A) 添加测试框架，使用 TDD 模式
- B) 不写测试，快速原型模式
- C) 只对关键路径写测试

## 5. 拆分任务

每个任务一个聚焦的目标，涉及尽量少的文件（通常 1-3 个）。

**TDD 模式（用户选 A 或 C，且任务可测试）：**
1. 写失败的测试
2. 跑测试确认失败
3. 写最小实现
4. 跑测试确认通过
5. 提交

**非 TDD 模式（用户选 B，或任务不适用测试——配置、脚手架、文档、样式）：**
1. 创建/修改文件
2. 验证结果（运行命令或检查输出）
3. 提交

任务排序：基础设施优先，标记可并行的独立任务。

### 示例任务

```markdown
### Task 3: 实现通知 SSE 端点  ⬜

**文件:**
- 创建: `src/api/notifications/stream.ts`
- 测试: `tests/api/notifications/stream.test.ts`

**Step 1: 写失败的测试**
（关键逻辑完整代码，样板简述）

**Step 2: 跑测试确认失败**
运行: `npm test -- tests/api/notifications/stream.test.ts`
预期: FAIL

**Step 3: 写最小实现**
（关键逻辑完整代码，样板简述）

**Step 4: 跑测试确认通过**
运行: `npm test -- tests/api/notifications/stream.test.ts`
预期: PASS
```

注意：每个 Task 标题末尾标记 `⬜`（待完成），execute-plan 执行后会改为 `✅`。

## 6. 代码详细度

- 关键接口/类型定义：写完整
- 核心业务逻辑：写完整
- 样板代码（imports、配置、常规 CRUD）：简要描述 + 关键行
- 目标：每个 Task 的代码部分不超过 50 行

## 7. 计划文档格式

保存到 `docs/plans/YYYY-MM-DD-<feature>.md`。如果有对应设计文档，`<feature>` 与设计文档 slug 保持一致。没有设计文档时，从任务描述推导 slug（3-5 个英文单词，小写，连字符）。

```markdown
# [功能名称] 实现计划

**目标:** 一句话
**架构:** 2-3 句话
**技术栈:** 关键技术
**设计文档:** docs/designs/xxx.md（如果有）或"无（基于口头需求）"
**新增依赖:** 列出需要引入的新依赖（名称、用途、许可证）
**测试模式:** TDD / 非 TDD / 仅关键路径

---

### Task 1: [组件名称]  ⬜
...
```

## 结束

计划写完后告知用户文件路径，并提示：
- 准备开始实现？输入 `/spec:execute-plan`
- 需要调整计划
- 就到这里

## 原则

- 精确的文件路径（用工具确认后再写入计划）
- 精确的运行命令和预期输出
- 计划中的代码是参考蓝图，不是最终实现
- DRY、YAGNI

## 链路

- **Called by:** brainstorm 的结束引导、用户直接调用
- **Next step:** `/spec:execute-plan`
- **Pairs with:** brainstorm（设计不清晰时回退）
