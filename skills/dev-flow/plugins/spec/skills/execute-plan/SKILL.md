# 执行实现计划

严格按照实现计划逐步执行。

**开始时简要说明你将要做什么。**

**语言：** 跟随用户的语言偏好，默认中文。

## When NOT to Use

- 小 bug 修复或单文件改动 — 不需要计划执行流程，直接改
- 还没有实现计划 — 先用 `/spec:write-plan` 制定计划
- 需求还不清晰 — 先用 `/spec:brainstorm` 探索设计

## IRON LAW（违反即停止）

1. **每个修改必须对应计划中的 Task 编号** — 没有编号的修改不做
2. **每个"已完成"声明必须附带命令输出证据** — 没有证据不算完成
3. **自救前必须保存状态（git stash 或 branch）** — 没有快照不动手
4. **批次之间必须报告并等待用户反馈** — 不跳过检查点
5. **安全底线不可突破：** `rm -rf`、`git push --force`、`git reset --hard`、`curl | bash`、修改 `.env`/密钥/CI 安全检查 → 遇到即阻塞并报告

## 防合理化

| AI 自我辩解 | 正确做法 |
|-------------|----------|
| "这个偏差很小，属于微调" | 如果改了 API 签名、加了函数、改了依赖版本 → 是偏差，必须报告 |
| "测试失败但我知道原因" | 知道原因 ≠ 已修复。走自救流程 |
| "用户说全部执行，不需要暂停" | 全部执行 = 测试全过不暂停，测试失败仍然暂停 |
| "自救流程太重，这个小问题直接修" | 小问题也要 git stash 再修。没有快照的修复 = 无法回滚 |
| "这个文件不在计划里但需要改" | 如果计划没提到 → 是偏差，报告后再改 |

## 主动性要求

- 完成一个 Task 后，主动运行测试并贴出输出，不只是说"已完成"
- 信息不足时，先用 Grep/Glob/Read 查找，只问用工具查不到的问题
- 所有"已完成""已通过"的声明必须附带命令输出作为证据

## 1. 启动检查

```bash
git branch --show-current
git log --oneline -1
ls docs/plans/ 2>/dev/null || echo "NO_PLANS_DIR"
```

## 2. 加载计划

- 如果用户指定了计划路径，读取它
- 如果没指定，检查 `docs/plans/` 下的文件：
  - 只有一个文件 → 使用它并告知用户
  - **有多个文件 → 列出所有文件让用户选择，不自动选**
- 如果没有计划 → 提示用户：输入 `/spec:write-plan` 先制定计划
- 检查计划文件中的 Task 标记：如果有 `✅` 标记的任务，报告"检测到已完成的任务，将从第一个 ⬜ 任务开始"
- 检查计划创建后是否有提交修改了计划中涉及的文件（`git log --since=<plan-date> -- <文件列表>`），如有则报告给用户确认

## 3. 审查计划

- 阅读完整计划
- 如果计划文档中有 **设计文档:** 字段（且不是"无"），同时读取设计文档作为参考
- 有疑问或发现问题 → 先提出来，等确认后再动手

## 4. 逐批执行

默认每批 3 个任务。用户可以说"全部执行"（自动模式：测试全过就不暂停）、"每 5 个暂停"、或"从 Task N 开始"来调整。

每个任务：
1. 按步骤执行
2. 运行验证命令，确认结果符合预期
3. 提交代码
4. **在计划文件中将该 Task 的 `⬜` 改为 `✅`**

### 允许的微调（不需要报告）
- 修正 import 路径
- 调整缩进/格式以匹配项目风格
- 修正明显的拼写错误

### 需要报告但可继续的偏差
- API 签名与计划不完全匹配但功能等价
- 需要添加计划中未提及的辅助函数
- 依赖版本与计划指定的不同，需要调整 API 用法（报告版本差异和调整内容）

### 必须阻塞的问题（自救流程失败后）
- 计划中的架构方向与现有代码冲突
- 缺少关键依赖且无法安装
- 测试持续失败且自救流程无法解决

## 5. 批次报告

一切顺利时用精简格式：

```
批次 2/4 完成（Task 4-6），全部通过。当前进度 6/10。
回复"继续"执行下一批，或提出修改意见。如需中断，下次从 Task 7 开始。
```

有问题时展开详细报告。用户回复"继续"、"好"、"嗯"、"ok" → 继续下一批。

## 6. 阻塞处理（先分析，再自救，再升级）

遇到问题时：**不要直接尝试修复。** 先分析相关文件理解问题的完整上下文，确认问题根因后再动手。

### 自救前置步骤

每次自救前记录当前状态：`git stash` 或 `git checkout -b rescue-attempt-N`。自救失败后回滚：`git checkout` 回主分支，丢弃自救改动。

### 第 1 次失败：换一种方式（最多 2 次不同尝试）

- 换一种实现路径、不同的库、不同的命令
- 不重复同一种方法
- 尝试前 git stash，失败后恢复

### 第 2 次失败：系统化自救（按顺序执行一遍，不循环）

1. **读错误** — 逐字读错误信息，不跳过任何细节
2. **搜索** — 用错误信息的关键词在项目代码和文档中搜索
3. **最简检查** — 路径对不对？依赖装了吗？版本匹配吗？权限够吗？
4. **反转假设** — 验证你假设存在的东西是否真的存在

如果自救过程中发现了更深层的问题（如架构冲突），直接升级，不继续探索。

### 第 3 次失败：升级给用户

回滚到自救前状态，然后报告：

```
## 阻塞

**位置:** Task N, Step M
**代码库状态:** 已回滚到自救前的状态

**自救过程:**
- 第 1 次: [尝试了什么] → [为什么失败]
- 第 2 次: [尝试了什么] → [为什么失败]

**已排除的可能性:**
- [列出确认不是原因的东西]

**最可能的原因:**
- [基于自救过程的判断]

你可以：
A) 提供修改指示，我在当前 task 中调整
B) 输入 /spec:write-plan 重新规划
C) 手动修复后告诉我继续
```

## 7. 全部完成后

- 运行完整测试套件，贴出输出
- 记录起始和结束 commit：

```bash
echo "起始 commit: $(git log --oneline --reverse --since=<计划开始时间> | head -1)"
echo "结束 commit: $(git log --oneline -1)"
```

- 报告最终结果，包含：
  - 计划文档路径
  - 起始 commit hash
  - 结束 commit hash
- 问用户：需要代码审查？输入 `/spec:code-review`

## 链路

- **Called by:** write-plan 的结束引导、用户直接调用
- **Next step:** `/spec:code-review`
- **Pairs with:** code-review（实现后审查）
