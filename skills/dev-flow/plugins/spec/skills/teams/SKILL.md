# Agent Teams

用多个独立 agent 从不同视角并行完成任务，支持多轮递进式分析。

**开始时简要说明你将要做什么。**

**语言：** 跟随用户的语言偏好，默认中文。

**和 /spec:code-review 的区别：** code-review 是单人全面审查，有固定流程、自动修复和冰山法则。agent-teams 适合需要多个独立视角的深度分析。可以先用 `/spec:code-review` 做基础审查，再用 agent-teams 做多角度分析。

## When NOT to Use

- 任务有顺序依赖时（agent-teams 是并行独立分析，不适合需要步骤串联的任务）
- 简单的单一维度审查 — 用 `/spec:code-review` 更高效
- 需要修改代码 — agent-teams 只分析不修改，修复用 `/spec:execute-plan`

## 适用场景

问用户一个问题确认任务类型：

```
你想让团队做什么？

A) 设计评审 — 多角度审视一个设计方案
B) 代码审查 — 多视角审查代码变更
C) 方案探索 — 多人独立提出不同方案
D) 调试诊断 — 并行验证多个假设
E) 安全审计 — 多维度安全检查
F) 性能分析 — 前端/后端/数据库多层分析
G) 迁移评估 — 风险/兼容性/工作量多维评估
H) API 设计评审 — 消费者/实现者/演进多视角
I) 测试策略 — 多种测试思维规划覆盖
J) 事故复盘 — 时间线/根因/预防多维度
K) 文档审查 — 准确性/完整性/新用户视角
L) 其他 — 描述你的需求
```

收集必要的上下文（读相关文件、diff 等）后进入团队组建。

## 团队组建原则

根据任务动态提出团队方案。

**团队规模：** 2-3 人为主，最多 4 人。更多 = 噪音，不是价值。

**角色设计：**
- 每个角色必须有**独特的视角**，不是同一个视角的重复
- 每个角色的审查维度必须**互不重叠**
- 每个角色的产出格式必须**明确定义**
- 角色名称使用具体的职能描述

呈现方案表格（角色 / 视角 / 重点关注 / 产出），等用户确认。

### 常见角色组合参考（不是模板，根据实际情况调整）

**设计评审：** 用户体验视角 / 技术可行性视角 / 挑战者视角

**代码审查：** 安全与数据视角 / 逻辑与边界视角 / 可维护性视角

**方案探索：** 激进方案 / 保守方案 / 挑战者

**调试诊断：** 假设 A 调查员 / 假设 B 调查员 / 环境与配置排查员

**安全审计：** OWASP 检查员 / 认证与授权流程分析 / 数据流与泄露追踪

其他场景（性能分析、迁移评估、API 评审、测试策略、事故复盘、文档审查等）根据任务类型动态设计角色。

## 并行派发

用 Agent 工具**同时** spawn 所有 agent（一条消息中发出所有 Agent 调用）。

**上下文裁剪：**
- 单个 agent prompt 内容超过 500 行：只粘贴与该 agent 审查维度相关的部分
- 不相关的部分用一句话摘要替代
- 裁剪后仍过大 → 告知用户并建议缩小范围

每个 agent prompt 必须自包含（角色定义 + 关注维度 + 不需要关注的 + 上下文 + 产出格式 + 语言）。

## 单轮汇总

所有 agent 返回后，汇总为统一报告：

```markdown
## Agent Teams 报告（第 N 轮）

### 任务 / 团队

---

### 角色 1 的发现
[核心发现，去掉冗余]

### 角色 2 的发现
[同上]

---

### 交叉分析

**共识：** [多个角色都提到的]
**独特发现：** [只有一个角色发现的]
**矛盾：** [意见不一致的 → 你的判断]

### 建议的行动项
1. [必须做]
2. [应该做]
3. [可选]
```

## 多轮执行

单轮汇总完成后，问用户：

```
第 1 轮完成。发现了 [N 个共识 + M 个独特发现 + K 个矛盾]。

下一步：
A) 启动第 2 轮 — 针对第 1 轮发现的重点问题，组建新团队深入分析
B) 就到这里 — 基于当前发现开始行动
```

### 第 2 轮及后续轮次的设计

如果用户选择继续：

**1. 识别深入方向**

从第 1 轮的交叉分析中提取需要深入的领域：
- 矛盾点（角色之间意见不一致的地方最值得深入）
- 高风险独特发现（只有一个角色发现但影响重大的）
- 共识中的表层结论（多人都提到但可能没深入挖的）

**2. 组建新团队**

第 2 轮的角色必须和第 1 轮**完全不同**——不是重复审查，是换角度深入。

第 2 轮角色必须和第 1 轮**完全不同**：第 1 轮是广度扫描，第 2 轮是针对发现的重点问题深度钻探（如从"用户体验视角"变为"新用户完整流程模拟"）。

**3. 将第 1 轮发现注入第 2 轮**

第 2 轮 agent 的 prompt 中必须包含：
- 第 1 轮的交叉分析摘要（让第 2 轮 agent 知道已经发现了什么）
- 第 1 轮中与该 agent 维度相关的具体发现
- "你的任务是在这些发现的基础上更深入，而不是重复第 1 轮的分析"

**4. 跨轮次汇总**

```markdown
## Agent Teams 最终报告

### 分析概要
- 第 1 轮: [N 人，广度扫描，发现 X 个问题]
- 第 2 轮: [M 人，深度钻探，深入了 Y 个领域]

### 最终发现（按优先级排序）

**🔴 必须立即处理**
1. [问题] — 第 N 轮 [角色] 发现，第 M 轮 [角色] 确认/深化

**🟡 应该处理**
2. [问题]

**🔵 建议关注**
3. [问题]

### 跨轮次洞察
[第 1 轮和第 2 轮结合后才能看到的模式或结论]
```

### 轮次限制

- 最多 3 轮。第 3 轮后如果仍有未解决的问题，列出并建议人工处理
- 每轮之间必须有用户确认，不自动启动下一轮
- 后续轮次的团队规模通常比第 1 轮小（聚焦深入，2 人足够）

## 后续引导

- 发现需要修复的问题 → 手动修复 或 输入 `/spec:execute-plan`
- 发现设计缺陷 → 输入 `/spec:brainstorm`
- 方案探索产出了值得保存的设计 → 在 `/spec:brainstorm` 中选保存
- 全部通过 → 继续下一步

## 原则

- 2-3 个高质量视角 > 5 个重叠的视角
- 交叉分析比单个报告重要
- agent prompt 必须自包含
- 不预设角色，每次根据任务动态决定
- 多轮的价值在于深度递进，不是广度重复
- 如果第 1 轮发现 < 3 个问题且无矛盾，建议直接结束

## 链路

- **Called by:** code-review 的审查后引导、用户直接调用
- **Next step:** `/spec:execute-plan`（修复问题）或 `/spec:brainstorm`（设计缺陷）
- **Pairs with:** code-review（先基础审查再多角度分析）
