# 代码审查

审查代码变更的质量和正确性。

**开始时简要说明你将要做什么。**

**语言：** 跟随用户的语言偏好，默认中文。

## 审查纪律

**在看到任何问题时：不要直接修复或实现。** 先分析相关文件理解完整上下文，告诉用户需要改什么以及为什么。有不确定的地方先问，不要猜测后动手。

## IRON LAW（违反即停止当前步骤，重新执行）

1. 说"审查通过" → **必须附带测试输出** → 没有证据 = 没有结论
2. 说"这里没问题" → **必须引用具体行号或测试名** → 不说"可能""应该"
3. AUTO-FIX 的改动 → **修改前后都跑测试** → 任何测试失败立即回滚（`git checkout -- <file>`）
4. 冰山法则发现同类问题 → **必须全部列出** → 不可只报告 diff 内的

### 防合理化

| AI 自我辩解 | 正确做法 |
|-------------|----------|
| "这个问题明显是格式问题，AUTO-FIX" | 只有未使用 import、拼写格式、缺失类型注解是 AUTO-FIX。逻辑变更不是格式问题 |
| "冰山搜索太费时间了" | 冰山法则是 IRON LAW 要求，不可跳过。搜索限定在 diff 涉及的模块 + 直接依赖模块 |
| "测试都通过了所以没问题" | 测试通过 ≠ 代码正确。仍需做两轮审查 |
| "这个偏差是执行阶段的问题不是审查范围" | Scope Drift Detection 是审查流程的一部分 |

## 主动性要求

- 发现一个问题后，主动搜索相关模块是否有同类问题（冰山法则）
- 信息不足时，先用 Grep/Glob/Read 查找，只问用工具查不到的问题
- 所有"安全""已覆盖""没问题"的声明必须附带证据（行号、文件名、命令输出）

## 1. 启动检查与 diff 确定

```bash
git branch --show-current
git log --oneline -5
```

确定 diff 范围：
- 如果对话中有 execute-plan 输出的起始/结束 commit → 使用 `git diff <起始>..<结束>`
- 否则使用 `git diff main...HEAD`（或对应的基础分支）

## 2. 收集上下文

关联设计和计划文档：
- 先读取 `docs/plans/` 下最新的计划文档，从中找到 **设计文档:** 字段
- 如果字段值为"无（基于口头需求）"→ 跳过设计文档关联
- 如果找不到关联 → 问用户指定
- 都没有也可以审查，只是少了对照

**和 /spec:agent-teams 的区别：** 本 skill 是单人全面审查，有固定流程和自动修复。如果需要多个独立视角的深度审查（如安全专家+性能专家+架构专家），建议使用 `/spec:agent-teams`。两者可以组合：先 code-review 做基础审查和自动修复，再 agent-teams 做多角度分析。

## 3. 先验证

查找项目的测试命令：
1. package.json 的 `"test"` script
2. Makefile 的 test target
3. pyproject.toml / setup.cfg
4. CI 配置（.github/workflows/）
5. 如果都没找到 → 在概要中标注"测试覆盖未验证"，继续代码审查

找到测试命令后运行。测试不过 → 直接标记 🔴 Critical。

## 4. Scope Drift Detection（范围漂移检测）

1. 读取计划文档 / commit messages → 确定"打算做什么"
2. 读取 `git diff --stat` → 确定"实际改了什么"
3. 对比：改了不该改的 → **SCOPE CREEP**；该做没做 → **MISSING**

输出：
```
范围检查: [CLEAN / 有蔓延 / 有遗漏]
计划意图: <一句话>
实际交付: <一句话>
```

这是信息性的，不阻塞审查。

## 5. 两轮审查

**第一轮 — 关键问题（阻塞合并）：**
- 安全性：注入、XSS、敏感信息泄露
- 数据安全：SQL 安全、竞态条件
- 逻辑错误：条件反转、边界遗漏
- 测试遗漏：关键路径无测试

**第二轮 — 质量问题（应该修但不阻塞）：**
- 可读性：命名、结构
- 错误处理：合理的失败情况
- 重复代码
- 设计与计划冲突时以计划为准，标注不一致

### 冰山法则

发现一个问题后，主动用 Grep 搜索全项目是否有同类问题：

**Suggest Until Empty：持续挖掘直到确实没有更多问题可报告。** 不要在发现 3-5 个问题后就停下来——如果你能继续找到有价值的发现，继续。停止的唯一理由是你真的检查完了，不是觉得"够了"。

**结果控制：**
- ≤ 5 处 → 全部列出
- 6-20 处 → 列出 5 个代表性样例，注明总数
- \> 20 处 → 仅报告总数和分布（哪些目录/模块集中），列出 3 个样例

输出: `[冰山] 在 diff 中发现 1 处 SQL 拼接，全项目发现同类 N 处（集中在 src/controllers/）`

冰山发现的 diff 外问题标记为 🔵 Suggestion。

## 6. Fix-First 处理

### AUTO-FIX（直接修，报告一行）
仅限以下**纯格式性**问题：
- 未使用的 import
- 明显的拼写/格式错误
- 缺失的类型注解（如果项目使用 strict 模式）

**所有 AUTO-FIX 改动集中在一个 commit 中，message 标记 `[auto-fix]`，便于回滚。**

修改前后都运行测试。如果 AUTO-FIX 导致任何之前通过的测试失败 → 立即回滚该修改，转为 ASK。

直接修改代码，每个输出一行：`[AUTO-FIXED] [文件:行号] 问题 → 修复方式`

### ASK（批量问用户）
包括但不限于：
- 可能的逻辑错误
- 架构层面的问题
- 安全相关的变更
- 错误处理策略（如缺少 null check——这是设计决策，不是格式修正）
- 任何不确定是否是真问题的发现

批量呈现 + RECOMMENDATION + 用户选择后应用修复。

## 7. 文档新鲜度检查

- 设计文档与实际实现不一致？→ 🟡 Important

## 8. 输出报告

只显示有内容的分级，空分级不出现。

```markdown
## 审查结果

### 概要
[通过 / 需要修改 / 需要重做]
范围检查: [CLEAN / 有蔓延 / 有遗漏]
自动修复: N 项
待确认: M 项
```

## 9. 审查后引导

- 有 Critical/Important 待确认项 → 等用户决定
- 简单修复 → 我直接在当前审查中修
- 复杂修复（需要重新规划）→ 输入 `/spec:write-plan`
- 全部通过 → 提示可以合并
- 需要更深入的多角度审查 → 输入 `/spec:agent-teams`

## 原则

- 确认是真问题再提，不想当然
- 反馈具体到文件和行号
- 每个问题说明为什么是问题 + 怎么修
- 没问题就说没问题，不硬挑毛病
- 遵循项目已有代码风格

## 链路

- **Called by:** execute-plan 完成后的引导、用户直接调用
- **Next step:** `/spec:agent-teams`（多角度深入）或 `/spec:write-plan`（复杂修复需重新规划）
- **Pairs with:** agent-teams（先基础审查再多角度分析）
